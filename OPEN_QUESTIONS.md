## Этап 1 — мок-слой

**Вопрос:** как выпускать карты ФЛ-резиденту при нескольких ТС одной категории (например, два легковых автомобиля)?
ТЗ (раздел 3) описывает лимит легковой категории как «100 л/чел.» (на человека), а грузовой — как «300 л/ТС» (на ТС), но явно не разбирает случай нескольких ТС одной категории у одного человека.
**Допущение:** персональная карта (легковая, 100 л) не дублируется независимо от числа легковых ТС — она одна на человека; для каждого грузового ТС выпускается отдельная карта (лимит закреплён за ТС). Реализовано в `src/lib/cardRules.ts`. Подлежит подтверждению заказчиком.

**Вопрос:** какой конкретно признак ИИН/БИН/паспорта/ГРНЗ должен детерминированно вызывать сценарий ошибки в мок-проверках (тех.план 2.3 говорит об этом в общем виде, не называя конкретное правило)?
**Допущение** (реализовано в `src/mocks/api/*.ts`, используется для демонстрации негативных сценариев в этапе 3):
- ИИН оканчивается на «9» → ГБД ФЛ: не найден; на «8» → ГБД ФЛ: ФИО не совпадает; на «7» → БМГ: телефон не привязан; на «6» → ОГПО/ЕСБД: не застрахован.
- БИН оканчивается на «9» → ГБД ЮЛ: не найден.
- Номер паспорта содержит `DUP` → «Беркут»: дубликат паспорта; содержит `NF` → «Беркут»: личность не подтверждена.
- ГРНЗ содержит `ERR` → ИС «Автомобиль»: ТС не найдено.
- SMS-код и код ЭЦП — фиксированный демо-код `1234` (подсказывается в data успешного `sendSmsCode`); любой другой ввод — ошибка неверного кода.

**Вопрос:** нужно ли ограничивать суточный лимит иностранца/ЮЛ-нерезидента числом (потолок отсутствует по ТЗ), но как тогда показывать прогресс-бар лимита (8.3) на их карте?
**Допущение:** для карт без льготного лимита (`dailyLimitL: null`) UI на этапах 2/5/6 покажет только «объём сегодня» без прогресс-бара/потолка — компонент `LimitProgressBar` должен поддерживать этот режим отдельно.
