# Регистрация в JanarmAI — как это устроено

Справочник по потребительской регистрации: какие есть пути, какие данные использовать для
тестов, как работает мок-логика проверок и выпуска карт. Все данные в проекте — мок (без
реального бэкенда), персистятся в `localStorage` браузера.

Дев-сервер: `npm run dev` → **http://localhost:5173**.

---

## 1. Обзор — 2 входные точки, 5 веток

Регистрация начинается на `/` (`ChannelSelectPage.tsx`, S-00) и предлагает два канала:

| Кнопка на `/` | Куда ведёт | Кто регистрируется |
|---|---|---|
| **«Через eGov / банковское приложение»** | `/register/egov-bvu` | Только ФЛ-резидент, самый короткий путь (данные уже подтянуты каналом) |
| **«Приложение КМГ»** | `/register/kmg` → выбор типа лица | ФЛ-резидент (полный путь), ФЛ-иностранец, ЮЛ-резидент, ЮЛ-нерезидент |

На `/register/kmg` (`KmgRegisterPage.tsx`) выбор типа лица ведёт дальше:

```
/register/kmg
 ├─ «ФЛ · резидент РК»     → остаётся на этой же странице (шаги встроены)
 ├─ «ФЛ · иностранец»      → /register/kmg/foreign
 ├─ «ЮЛ · резидент РК»     → /register/kmg/company
 └─ «ЮЛ · нерезидент»      → /register/kmg/company-foreign
```

Итого **5 законченных сценариев регистрации** — каждый заканчивается вызовом
`finalizeFlRegistration` (ФЛ) или `finalizeUlRegistration` (ЮЛ) из
`src/features/onboarding/registrationActions.ts`, который:
1. создаёт пользователя/компанию в `user.store.ts`,
2. выпускает карту(ы) по правилам `src/lib/cardRules.ts`,
3. **сразу логинит** созданный аккаунт (автологин — см. раздел 4),
4. переносит на `/card` (ФЛ) или `/cabinet` (ЮЛ) с баннером «Карта успешно выпущена».

## 4. Ключевая бизнес-логика

### 4.1. Автологин после регистрации

`finalizeFlRegistration`/`finalizeUlRegistration` в конце вызывают `login()` — пользователь
сразу оказывается в сессии и попадает на защищённый маршрут (`/card` или `/cabinet`) без
отдельного экрана «выпуск карты». Раньше это было пропущено (баг), сейчас — обязательная
часть завершения регистрации.

### 4.2. Какая карта выпускается — `src/lib/cardRules.ts`

| Кто | Лимит/сутки | Цена | Логика |
|---|---|---|---|
| ФЛ-резидент, легковая или без своего ТС | 100 л | льготная | Одна карта **на человека** — не дублируется, если легковых ТС несколько |
| ФЛ-резидент, грузовая | 300 л | льготная | Отдельная карта **на каждый грузовик** |
| ФЛ-иностранец | без лимита | предельная | Одна карта, льгота не действует |
| ЮЛ-резидент | 100 л (легковая) / 300 л (грузовая) | льготная | Карта **на каждое ТС автопарка** |
| ЮЛ-нерезидент | 100 / 300 л (лимит есть) | предельная | Как ЮЛ-резидент по объёму, но без льготной цены |

### 4.3. Единый запрос к базе МВД — `src/mocks/api/vehicle.ts` (`checkMvdRegistry`)

Раньше было 2 отдельных мока (ТС + ОГПО), сейчас один вызов по ИИН/паспорту определяет и то,
и другое одним ответом. Категория ТС и наличие полиса — по последней цифре идентификатора
(см. таблицу триггеров ниже). ГРНЗ генерируется автоматически, вручную не вводится.

### 4.4. Согласие на обработку персональных данных

Только в ветке eGov/БВУ — чекбокс на экране с уже показанными данными (`identity`-шаг),
блокирует кнопку «Продолжить», пока не отмечен. В ветке КМГ согласие пока не размещено
(отдельное решение, ждёт уточнения — см. `OPEN_QUESTIONS.md`).

## 5. Пять сценариев по шагам

### 5.1. eGov / БВУ — ФЛ-резидент (самый короткий)

`/` → «Через eGov / банковское приложение» → `/register/egov-bvu`

1. **Данные подтверждены каналом** — ФИО/ИИН/телефон генерируются случайно при каждом заходе
   (`generateDemoFio/Iin/Phone` из `src/lib/demoIdentity.ts`), намеренно «чистые» (последняя
   цифра ИИН — 0–5, никогда не попадает под коды ошибок). Нужно отметить чекбокс согласия →
   «Продолжить».
2. Авто-степпер `checkMvdRegistry(iin)` — без ручного ввода.
3. Итоговая карточка лимита → «Выпустить карту» → `/card` с баннером.

### 5.2. Приложение КМГ — ФЛ-резидент (полный путь, ручной ввод)

`/` → «Приложение КМГ» → «ФЛ · резидент РК» (остаётся на `/register/kmg`)

1. Форма: ФИО, ИИН (12 цифр), телефон (`+7XXXXXXXXXX`) — **вводится вручную**, поэтому здесь
   можно намеренно спровоцировать любую ошибку из таблицы ниже.
2. Степпер ГБД ФЛ → БМГ (`checkGbdFl` → `checkBmg`).
3. Степпер `checkMvdRegistry`.
4. Итог → «Выпустить карту» → `/card`.

### 5.3. ФЛ-иностранец

`/register/kmg` → «ФЛ · иностранец» → `/register/kmg/foreign`

1. Номер паспорта + «Сканировать паспорт» (MRZ, мок) + «Пройти liveness» (селфи, мок).
2. Степпер «Беркут» (`checkBerkut`) — может отказать (см. таблицу).
3. Телефон — просто контакт, без SMS.
4. Степпер `checkMvdRegistry` (по номеру паспорта).
5. Итог — **всегда «без лимита», предельная цена** → «Выпустить карту» → `/card`.

### 5.4. ЮЛ-резидент (автопарк)

`/register/kmg` → «ЮЛ · резидент РК» → `/register/kmg/company`

1. БИН компании + ФИО представителя + телефон → код ЭЦП (`1234`).
2. Степпер ГБД ЮЛ (`checkGbdUl`) по БИН → подтягивает название компании.
3. Добавление ТС автопарка (ГРНЗ + категория, можно несколько).
4. Назначение водителей — опционально, не влияет на лимит.
5. «Продолжить» → `/cabinet` с баннером — по карте на каждое добавленное ТС.

### 5.5. ЮЛ-нерезидент

`/register/kmg` → «ЮЛ · нерезидент» → `/register/kmg/company-foreign`

1. Ручной ввод компании (название, рег. номер, телефон) — **без** проверки ГБД ЮЛ.
2. Паспорт + liveness + «Беркут» для директора-иностранца (те же экраны, что в 5.3).
3. Автопарк + водители — как в 5.4.
4. «Продолжить» → `/cabinet` — карты есть (по объёму как у резидента), но **без льготной цены**.

## 6. Шпаргалка: чем спровоцировать ошибку (только там, где ввод ручной — 5.2/5.3/5.4)

| Проверка | Триггер | Результат |
|---|---|---|
| ГБД ФЛ | ИИН оканчивается на `9` | «не найден в ГБД ФЛ» |
| ГБД ФЛ | ИИН оканчивается на `8` | «ФИО не совпадает с ИИН» |
| БМГ | ИИН оканчивается на `7` | «телефон не привязан к ИИН» |
| МВД (ТС+ОГПО) | ИИН/паспорт оканчивается на `6` | «Не удалось получить данные из базы МВД» |
| МВД (ТС+ОГПО) | оканчивается на `0` | нет своего ТС, есть ОГПО → карта 100 л |
| МВД (ТС+ОГПО) | чётная (кроме 0/6) | своя легковая → 100 л |
| МВД (ТС+ОГПО) | нечётная (кроме 6) | своя грузовая → 300 л |
| Беркут | паспорт содержит `DUP` | «дубликат паспорта» → отказ в регистрации |
| Беркут | паспорт содержит `NF` | «личность не подтверждена» → отказ |
| ГБД ЮЛ | БИН оканчивается на `9` | «не найден в ГБД ЮЛ» |
| SMS / ЭЦП код | любой, кроме `1234` | «Неверный код подтверждения» |

Ветка 5.1 (eGov) использует автогенерируемые «безопасные» данные — там эти триггеры не
воспроизвести без правки кода, только 5.2/5.3/5.4 дают ручной ввод.
