## Этап 0 — Инициализация проекта

**Сделано:**

Отмечено как замена в рамках допустимых равнозначных решений (см. правила AGENT_BUILD_PLAN.md).
- `src/index.css` — токены палитры (белый / navy / orange) по разделу 8.1 ТЗ через директиву `@theme` (шкалы navy/orange 50–950, семантические токены `--color-limit-normal/warning`, `--color-status-ok/blocked`).

  - `src/admin/routes.tsx` + `src/admin/layout/AdminLayout.tsx` — аналитический модуль, desktop-only (`min-w-[1280px]`), отдельный визуальный стиль (тёмный navy-фон, шапка «JanarmAI · Аналитика»).
  - `src/AppRouter.tsx` — корневой `BrowserRouter`, разводит `/admin/*` и `/*`.
- Заглушки (`RouteStub`) на все маршруты из карты маршрутов раздела 4 тех.плана: `/`, `/register/egov-bvu`, `/register/kmg`, `/register/kmg/foreign`, `/register/kmg/company`, `/register/kmg/company-foreign`, `/login`, `/card`, `/cabinet`, `/cabinet/history`, `/admin/login`, `/admin/dashboard` + catch-all 404 в обоих деревьях.

## Этап 1 — Мок-слой

Допущение по неоднозначности ТЗ зафиксировано в `OPEN_QUESTIONS.md`.
- `src/lib/{id,time,mask}.ts` — генерация id/QR-токенов, расчёт ближайшей полуночи по времени Астаны (сброс лимита), маскирование ИИН/БИН.
- `src/mocks/seed/*.ts` — стартовые данные: 5 пользователей (по одному на каждую ветку регистрации/входа: ФЛ-резидент eGov, ФЛ-резидент КМГ, ФЛ-иностранец, директор ЮЛ-резидента, директор ЮЛ-нерезидента), 2 компании, 8 ТС, 3 водителя, справочник лимитов/цен (100/300 л, мок-цены льготная/предельная по видам топлива), 15 АЗС по областям РК (с пометкой приграничных с РФ регионов — под будущую тепловую карту Этапа 7), 3 служебных аккаунта аналитиков, карты (сгенерированы через `cardRules` из seed-пользователей/ТС/компаний), стартовая история заправок (11 транзакций).
- `src/mocks/api/*.ts` — имитация проверок: `checkGbdFl`, `checkBmg`, `sendSmsCode`/`verifySmsCode` (identity.ts), `checkBerkut`/`verifyLiveness` (foreigner.ts), `checkVehicleRegistry`/`checkOgpoInsurance` (vehicle.ts, ЕСБД/ОГПО), `checkGbdUl`/`signEcp` (company.ts), `checkAdminCredentials` (admin.ts). Общий формат ответа `{status:'success', data}` / `{status:'error', errorCode, message}` через `mockSuccess`/`mockError` (`mockRequest.ts`), задержка 300–1500 мс (`config.ts`). Детерминированные сценарии ошибок — по признаку ИИН/БИН/паспорта/ГРНЗ, правило зафиксировано в `OPEN_QUESTIONS.md`.
- `src/store/*.ts` — 4 Zustand-стора с `persist` в localStorage: `user.store.ts` (пользователь/сессия + рантайм-база users/companies/vehicles/drivers, пополняемая при регистрации), `card.store.ts` (карты и лимиты: `consumeLimit`, `resetDaily`, `regenerateQrToken`), `transaction.store.ts` (история), `admin.store.ts` (админ-сессия). `resetDemoData.ts` — сброс всех сторов к seed по кнопке «Сбросить демо-данные» (будет подключена в UI на Этапе 6/8).
- `src/dev/MocksDevPage.tsx` на маршруте `/dev/mocks` — визуальная проверка мок-вызовов и состояния сторов вне бизнес-потоков (по аналогии с `/dev/components` из Этапа 2).
- `scripts/smoke-mocks.ts`, `scripts/smoke-persist.ts` + npm-скрипты `smoke:mocks`/`smoke:persist` (через `tsx`) — дымовые скрипты для проверки мок-слоя и persist-слоя из консоли без браузера.

**Как проверено:**
- `npx tsc -b --noEmit` и `npm run build` — без ошибок.
- `npm run smoke:mocks` — все 10 проверок из консоли вернули корректный `{status, data|errorCode, message}`, включая все запланированные негативные сценарии (ГБД ФЛ не найден/несовпадение, БМГ, SMS неверный код, «Беркут» дубликат, ИС «Автомобиль» не найдено, вход администратора неверный); суммарная задержка 11 вызовов ≈9с, что соответствует диапазону 300–1500мс на вызов.
- `npm run smoke:persist` — с полифиллом `localStorage` в Node подтверждено: (1) `consumeLimit` корректно обновляет карту в сторе, (2) обновлённое состояние физически записывается в `localStorage` под ключами `janarmai-card-store`/`janarmai-admin-store` (проверено содержимое сериализованной записи), (3) `reset()` возвращает сторы к seed-состоянию. Это подтверждает контракт persist-слоя, лежащий в основе требования «состояние переживает перезагрузку страницы» — сам факт перезагрузки браузера дополнительно не тестировался (headless-браузер недоступен в этом окружении), библиotека zustand/persist общепринята и стабильна.
- Dev-сервер (`npm run dev`) поднят, маршрут `/dev/mocks` отдаёт 200.

**Отклонения/примечания:**
- Установка `tsx` (как devDependency для дымовых скриптов) подтянула транзитивную уязвимость high severity (ReDoS в `d3-color` через `react-simple-maps`, `npm audit`). Не исправлено намеренно: fix ломает `react-simple-maps` до v1 (несовместимо с тех.планом, который явно требует эту библиотеку для карты РК), а сама уязвимость не эксплуатируема в контексте демо со статичными встроенными данными (нет обработки пользовательского ввода через уязвимый регэксп).

## Постфактум-правка №2 — исправление тепловой карты (по скриншоту пользователя после деплоя)

Пользователь задеплоил приложение на Vercel и прислал скриншот: `/admin/dashboard` рендерился сплошным тёмно-синим прямоугольником без единой видимой границы региона — то есть визуальная проверка, недоступная в этом окружении на всех предыдущих этапах, впервые стала возможна, и подтвердила ранее задокументированный риск (`OPEN_QUESTIONS.md`: «проекция/масштаб не проверялись визуально»).

**Сделано:**
1. **`vercel.json`** — добавлен rewrite всех путей на `/index.html`. Без него Vercel возвращал `404 NOT_FOUND` на любой прямой переход/обновление по клиентскому маршруту (например `/admin/login`) — SPA-роутинг целиком на стороне React Router, а Vercel по умолчанию ищет физический файл.
2. **`KzHeatMap.tsx`** — вручную подобранные `center=[67,48]`/`scale=1500` заменены на самовписывающуюся проекцию `geoMercator().fitExtent(...)` (`d3-geo`), которая вычисляет масштаб/центр из фактического bounding box геоданных — устраняет саму категорию проблемы (не нужно вручную «на глаз» подбирать числа без визуальной обратной связи).
3. **Найден и исправлен более серьёзный скрытый баг**, вскрывшийся при диагностике: после конвертации шейпфайла через `mapshaper` порядок обхода вершин (winding order) у **всех 20** полигонов был инвертирован — `d3-geo` интерпретировал каждый регион как «весь земной шар минус сам регион» (`geoArea` ≈ 4π ср, `geoBounds` = вся сфера). Это и есть настоящая причина «сплошного прямоугольника»: даже с идеально подобранным `scale`, карта рендерила бы то же самое, потому что рендерился не Казахстан, а его инверсия. Флаг `mapshaper ... rfc7946` не устранил проблему в установленной версии инструмента — исправлено вручную реверсом порядка точек в каждом кольце координат.
4. `scripts/smoke-heatmap-projection.ts` (`npm run smoke:heatmap`) — новый дымовой тест, который теперь ловит именно этот класс ошибок без браузера: считает пиксельные bbox всех регионов через `d3-geo geoPath.bounds()` и падает, если (а) координаты не число, (б) какой-то регион занимает >90% холста (сигнатура бага winding-инверсии), (в) страна обрезана или, наоборот, не заполняет холст. Добавлен в `smoke:all`.

**Как проверено:** диагностический прогон подтвердил проблему (`geoBounds` всех 20 регионов = `[[-180,-90],[180,90]]`, `geoArea` ≈ 12.56 ср = 4π у каждого — то есть буквально «вся сфера» для каждого региона одновременно, что и объясняет сплошную заливку на скриншоте); после реверса колец — `geoBounds` стали корректными (лон 46.49–87.28°E, лат 40.59–55.43°N, в точности контур Казахстана), `smoke:heatmap` проходит (страна вписывается в холст 800×480 без обрезки, 0 вырожденных регионов). Полная регрессия (`tsc`, `build`, `lint`, `smoke:analytics`) — без ошибок.

**Урок на будущее (зафиксирован в `OPEN_QUESTIONS.md`):** при любой будущей смене источника геоданных для карты — обязательно прогонять `npm run smoke:heatmap`, а не полагаться на успешную сборку: TypeScript и Vite физически не могут поймать некорректную геометрию (JSON синтаксически валиден), инверсию winding-order ловит только сама геометрическая библиотека при вычислении площади/границ. Также это прямое практическое подтверждение риска, о котором предупреждалось во всех предыдущих PROGRESS.md-записях: без визуальной проверки в браузере часть багов остаётся невидимой до первого реального просмотра пользователем — что здесь и произошло.

## Постфактум-правка №3 — реальный runtime-баг react-simple-maps (найден по консоли браузера после деплоя)

После правки winding order пользователь передеплоил и всё равно увидел белый экран на `/admin/dashboard`, с ошибкой в консоли хостинга: `TypeError: r is not a function` внутри минифицированного `useMemo`. Попытка воспроизвести локально; сеть в среде временно отсутствовала — пользователь восстановил её и попросил продолжить.

**Диагностика (без слепого исправления):**
- Прочитан исходник `node_modules/react-simple-maps/dist/index.es.js` (`makeProjection`): при передаче функции в проп `projection`, библиотека делает `if (typeof projection === 'function') return projection` — то есть **не вызывает** её как фабрику `(width, height, config) => GeoProjection`, вопреки декларации в `@types/react-simple-maps`. Переданная функция затем используется напрямую как d3-объект проекции: `geoPath().projection(proj)`, и далее `proj([lon, lat])` вызывается для каждой точки геометрии.
- `KzHeatMap.tsx` передавал именно фабрику (`useKzProjection`-функцию вида `(width, height) => GeoProjection`). При вызове `proj([lon, lat])` первым аргументом «шириной» оказывался массив `[lon, lat]`, что и приводило к `TypeError: r is not a function` где-то внутри d3-geo при попытке использовать некорректно сконструированный результат.

**Исправлено:** проекция теперь вычисляется один раз через `useMemo` (`useKzProjection()`) и передаётся в `ComposableMap` уже готовым d3-объектом (не фабрикой) — с `as never`-каст и комментарием, объясняющим расхождение между типами и рантаймом этой версии библиотеки.

**Впервые в проекте — реальная визуальная проверка в браузере:** сеть восстановилась, установлен Playwright + Chromium (через `npm ... playwright install chromium`, ~300 МБ, в отдельной scratch-директории вне репозитория). Написан драйвер-скрипт: логин администратора → `/admin/dashboard` → скриншот + сбор `console`/`pageerror`. Результат: **0 ошибок**, дашборд отрисовался полностью, тепловая карта показывает настоящий узнаваемый контур Казахстана с видимыми границами 20 областей и корректной цветовой шкалой. Дополнительно прогнаны все 10 маршрутов потребительского модуля (мобильная ширина 390px) — тоже 0 ошибок консоли на каждом; скриншотами подтверждён визуально корректный вид экрана выбора канала, входа и списка типов регистрации.
Замеченный на скриншоте графика «СНТ vs Продажи» артефакт (две точки, похожие на слипшуюся «восьмёрку» справа от графика) при проверке через DOM (подсчёт `<circle class="recharts-dot">`, ровно 24 = 12 месяцев × 2 серии, без дублей) оказался не багом — это последние точки обеих серий за «Июл», близкие по значению (индекс легальности к концу периода 94%), визуально слипшиеся в сжатом PNG. Изменений не потребовалось — важный урок: проверять DOM/данные, а не чинить то, что показалось странным на скриншоте.

**Как проверено:** `npx tsc -b --noEmit`, `npm run build`, `npm run lint`, `npm run smoke:all` (все 9 дымовых скриптов) — без ошибок. `scripts/smoke-heatmap-projection.ts` дополнен проверкой именно этого соглашения о вызове: `path.projection()([lon,lat])` должен вернуть `[x,y]` (числа), а не объект/исключение — тест теперь ловит регресс этого класса без браузера в будущем. Playwright-проверка (вне репозитория, разовая) — реальный скриншот дашборда и всех экранов потребительского модуля, 0 ошибок консоли.

**Отклонения/примечания:** Playwright и Chromium установлены во временную scratch-директорию (`%TEMP%/pwdriver`, `%TEMP%/npmcache-scratch`) вне репозитория — не добавлены в зависимости проекта, так как это разовая диагностика в конкретной среде агента, а не часть постоянного тестового стека проекта (браузерное тестирование не входило в объём AGENT_BUILD_PLAN.md). Если нужна регулярная браузерная регрессия — стоит осознанно добавить `@playwright/test` в `devDependencies` отдельной задачей.

### Найденный и исправленный баг

При первой живой проверке (Playwright) на `/register/kmg` React выдавал предупреждение `Function components cannot be given refs` из `IinPhoneFormStep` → `Input`. Причина: примитивы `Input`/`Textarea`/`Button` были написаны как простые функциональные компоненты (актуальный паттерн shadcn для **React 19**, где `ref` — обычный проп), а проект на **React 18.3**, где функциональному компоненту `ref` без `React.forwardRef` не передаётся. `react-hook-form`'овский `register('fio')` возвращает `{ ref, onChange, onBlur, name }` — без корректного `ref` библиотека не может, например, вызвать `setFocus`/фокусировать первое невалидное поле. Исправлено: `Input`, `Textarea`, `Button` переписаны через `React.forwardRef` с `displayName`. Повторная проверка — предупреждение исчезло, 0 ошибок консоли на всех проверенных маршрутах.
